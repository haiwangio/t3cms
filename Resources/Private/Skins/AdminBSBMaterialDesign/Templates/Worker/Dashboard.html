Why we count on FLUX!

The content must be handled seperated from the way, how to configure the website or the theme. Short: We want to provide new content elements only via an extra extension for that. So, we were thinking about to use mask, but the workflow with mask was not effizient enougth. The mask_export made mask to a better option, but with mask_export we loose the flexibility for changing/adding/removing fields.. With FLUX fluidcontent elements, we just need one HTML file to create a content element. So the workflow is effizient enougth and the flexibility for fields is given, because all configuration is done in this one HTML file. The last big reason is, that FLUX can create Content + IRRE + Grid-Elements

Your backend module can look like this!

Just use layoutRootPaths.10 to change the layoutRootPath in your Extension and use the T3CMS Backend Module Layout:

# TypoScript SETUP
module.tx_yourcoolextension {
    view {
        layoutRootPaths.0 = EXT:your_cool_extension/Resources/Private/Layouts/
        layoutRootPaths.10 = EXT:t3cms/Resources/Private/Skins/AdminBSBMaterialDesign/Layouts/
    }
}

And use the BeDefault layout in your html templates:

<f:layout name="BeDefault" />

Check out the static demo to explore, what elements are available with this be module layout. Open static demo. Note: The t3cms backend module layout is a very slim fork of the original template.
